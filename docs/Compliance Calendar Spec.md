## 2. Check Definitions by Cadence

### QUARTERLY (Every 3 Months)

| ID | Name | PCI 4.0.1 Req | Conditional | Evidence Required |
|----|------|---------------|-------------|------------------|
| Q1 | External ASV Vulnerability Scan | **11.3.2** | No | Passing ASV report, ASV attestation, remediation evidence for findings |
| Q2 | Internal Vulnerability Scan | **11.3.1** | No | Authenticated scan output, risk-ranked findings, rescan confirmation for high/critical |
| Q3 | Wireless Scan — Rogue AP Detection | **11.2.1** | `has_wireless_in_cde` | Scan results, no unauthorized APs confirmed or remediated, AP inventory current |
| Q4 | Third-Party / Remote Access Account Review | **8.2.7** | No | Remote access account list, active/inactive confirmation, terminated accounts removed |
| Q5 | Payment Page Integrity Attestation | **11.6.1** | `has_ecommerce` | See note below |
| Q6 | Log Review Process Attestation | **10.4.1** | No | See note below |

**Q8 — 11.6.1 Payment Page (Continuous + Quarterly Attestation):**
The standard requires a *change and tamper detection mechanism* to be active continuously on payment pages. The quarterly check here is an attestation that (a) the mechanism was active and functioning all quarter, (b) all alerts were investigated and resolved, and (c) no unauthorized script changes occurred. Evidence = monitoring tool output for the quarter + sign-off.

**Q9 — Log Review (10.4.1 Daily, Quarterly Attestation with SIEM):**
Requirement 10.4.1 mandates daily review of all security events, CHD system logs, critical component logs, and security function server logs. There is no carve-out in the standard for SIEM — daily review is required regardless of tooling. A SIEM makes it operationally feasible; it does not reduce the cadence. The quarterly attestation here captures evidence that: (a) the SIEM/log review process was active all quarter, (b) alert rules are current, (c) all triggered alerts were investigated and documented, and (d) no gaps in log collection occurred. Without a SIEM (`siem_in_use = false`), display a warning that daily manual review must be documented separately.

### SEMI-ANNUAL (Every 6 Months)

| ID | Name | PCI 4.0.1 Req | Conditional | Evidence Required |
|----|------|---------------|-------------|------------------|
| S1 | NSC / Firewall Configuration Review | **1.2.7** | No | Ruleset export, review sign-off, justification for all rules, rules with no business justification removed |
| S2 | WAF Rule Review | **1.2.7** | `has_waf` | WAF config export, rule effectiveness review, false positive log, unused rules removed |
| S3 | Cloud NSG / Security Group Review | **1.2.7** | `has_cloud_infra` | NSG export (Azure/AWS/GCP), review sign-off, unused/overly permissive rules removed |
| S4 | All User Account Access Review | **7.2.4** | No | Full user + TPSP account list, review sign-off, inappropriate access removed, management acknowledgment |
| S5 | CDE Scoping Review & Attestation | **12.5.2** (all) / **12.5.2.1** (SP: required semi-annual) | No | Updated scoping diagram, data flow review, any scope changes documented |
| S6 | TPSP / Service Provider Scoping Analysis | **12.8.1, 12.8.2, 12.8.5** | No | TPSP inventory, CHD data flow per provider, responsibility matrix current, agreements reviewed |
| S7 | Encryption at Rest Review & Attestation | **3.4.1, 3.5.1** | No | See CRITICAL NOTE below |
| S8 | Segmentation Testing (SERVICE PROVIDERS ONLY) | **11.4.6** | `is_service_provider` | Pen test results confirming segmentation effective, scope doc, tester qualifications |

**Note on S1 — 7.2.4:**
Exact requirement text: "All user accounts and related access privileges, including third-party/vendor accounts, are reviewed **at least once every six months**." This includes accounts used by employees, contractors, consultants, and vendors. Management must acknowledge that access remains appropriate.

**Note on S2 — Scope for all entities:**
Requirement 12.5.2 applies to all entities but does not specify a cadence — it says "at least once every 12 months and upon significant change." However, 12.5.2.1 requires service providers to confirm scope **at least every six months**. We implement S2 as semi-annual for all entities because scope drift discovered late is more costly, and a semi-annual review satisfies both requirements. Mark as semi-annual; note the SP requirement is mandatory, others recommended.

**Note on S8 — 11.4.5 vs 11.4.6 Segmentation Testing:**
- **11.4.5** applies to ALL entities: "at least once every 12 months and after any changes." → This is an ANNUAL check (see A3 below).
- **11.4.6** applies to SERVICE PROVIDERS ONLY: "at least once every six months and after any changes." → S5 above is this SP-specific additional requirement.
Both require the same methodology — it's the same test done twice as often for SPs.

**CRITICAL NOTE — S4 Encryption at Rest (3.4.1):**

Disk-level encryption (BitLocker, FileVault, Azure Disk Encryption, AWS EBS encryption) does **NOT** satisfy PCI-DSS 3.4.1. PAN must be unreadable anywhere it is stored, which means:

- **Databases:** TDE (Transparent Data Encryption) or column-level encryption is required. No exceptions. Full-disk encryption of the database server is insufficient because anyone with OS-level access can read unencrypted data through the running database process. TDE encrypts the physical files so raw access yields only ciphertext.
- **Files / Documents:** PAN stored in any file format (CSV, spreadsheet, PDF, log, backup) must be PGP-encrypted or equivalent field-level encryption. A spreadsheet of PANs on an encrypted drive does not comply.
- **Backups:** Backup files containing PAN must be encrypted independently of the disk they reside on, using a key separate from the primary encryption key.
- **Display surfaces:** Anywhere PAN is rendered (UI, report, receipt, log output), only the first 6 and last 4 digits may be visible. Unmasking PAN in any context requires documented business justification, role-based access control authorization, and an auditable approval record. No exceptions.
- **Key management (3.5.1, 3.7.x):** Encryption keys must be stored separately from the data they protect. Key-encrypting keys must use split knowledge and dual control. Key custodians must be documented. Rotation schedules must be defined and followed.

Evidence for S4 attestation:
- Database TDE configuration confirmation (config export + screenshot showing encryption enabled)
- File encryption inventory (list of all locations where PAN is stored in files + encryption method per location)
- Backup encryption confirmation (separate key confirmed)
- PAN masking verification for all display surfaces (UI, reports, logs)
- Any active unmasking authorizations with business justification on file
- Key custodian list, rotation log, split-knowledge confirmation
